# `_common` — shared normative rules for the format specs

> Status: stable.
> This is the shared **published** source for rules that several format
> specs depend on, relocated verbatim from internal (unpublished) ADRs so
> an external reader never has to consult them. Where a format spec
> references "the shared emit invariants", "the patch slot / sentinel
> rule", "the integrity model", or the "reserved vocabulary", that
> reference resolves here. Self-contained: no ADR knowledge is assumed.

This document is normative. A per-format spec under
[`spec/formats/`](./) MAY narrow or extend a rule for its own version; on
any conflict, the per-format spec wins for that format, and the divergence
is stated explicitly in that spec.

---

## §1 yarn-berry emit invariants (version-invariant)

These rules govern `stringify(graph, options?)` for **every** yarn-berry
`yarn.lock` schema version (v3–v10 inclusive). They are version-invariant:
an individual yarn-berry-vN spec records only the field-level diffs of its
schema, and inherits everything below. The crown-jewel **SYML quoting
predicate** ([§1.5](#15-quoting-the-syml-quoting-predicate)) lives here
once precisely so it never drifts across the family.

> **Scope.** §1 is the **yarn-berry-family** emit contract — it is
> SYML- and `__metadata`-specific and applies to the yarn-berry family
> (v3–v10) only. npm / pnpm / bun / yarn-classic document their **own**
> emit inline in their respective specs and reference this document only
> for [§2](#2-patch-slot--tarballkey-sentinel) (patch slot),
> [§3](#3-integrity-model) (integrity), and
> [§4](#4-reserved-vocabulary) (reserved vocabulary).
>
> **Completion phases.** The *modify* / *enrich* / *optimize* phases moved
> out of the conversion-era ADR-0016 to the published
> [ADR-0023](../decisions/0023-graph-modification-and-completion.md)
> (modify + enrich) and
> [ADR-0024](../decisions/0024-optimize-phase.md) (optimize); sibling
> specs redirect there rather than re-deriving them.

### 1.1 The stringify entry point and the round-trip property

```ts
export interface YarnBerryStringifyOptions {
  lineEnding?: 'lf' | 'crlf'   // default: 'lf'
  cacheKey?:   string          // re-emitted into __metadata.cacheKey verbatim
}

export function stringify(graph: Graph, options?: YarnBerryStringifyOptions): string
```

The output **MUST** satisfy the *Graph-level roundtrip* property:

```
parse(stringify(parse(x))) ≡ parse(x)
```

— evaluated at the `Graph` level via the existing `Graph.diff(other)`
accessor (`addedNodes`, `removedNodes`, `changedNodes`, `addedEdges`,
`removedEdges` all empty; tarball map byte-equal under canonical
iteration; diagnostics ignored). Whether the stringify output is
byte-identical to `x` is **irrelevant** to this contract.

### 1.2 Canonical preamble

Regenerate the yarn-berry preamble verbatim:

```
# This file is generated by running "yarn install" inside your project.
# Manual changes might be lost - proceed with caution!
```

— followed by one blank line, then `__metadata:`. Source comments are
not preserved across roundtrip (see [Non-goals](#16-non-goals-explicit)).
The preamble is regenerated even if the input lacked one.

### 1.3 Block ordering

`__metadata` first, then entry blocks sorted by entry-key string under
`cmpStr` — the same comparator that pins graph iteration (canonical
content-sorted iteration order; see [§4](#4-reserved-vocabulary)).
Multi-spec entry keys (`"foo@npm:^1, foo@npm:^2"`) sort on the **joined
key string**, not on any one constituent spec.

### 1.4 Entry-internal field schedule

Fixed schedule, matching yarn's emitter output:

```
version, resolution, dependencies, optionalDependencies,
peerDependencies, peerDependenciesMeta, dependenciesMeta,
bin, linkType, languageName, conditions, checksum
```

Fields not present on the graph are absent. Sub-blocks (`dependencies`
etc.) sort their entries alphabetically by name.

### 1.5 Quoting — the SYML quoting predicate

A SYML key or value scalar **MUST** be double-quoted iff at least one of
the following five conditions holds. The same predicate applies to keys
and values; there is **no key-vs-value asymmetry**.

1. **Special character.** The scalar contains any of: space
   (` `, U+0020), tab (`\t`, U+0009), `:`, `,`, `[`, `]`, `{`,
   `}`, `#`, `&`, `*`, `!`, `|`, `>`, `'`, `"`, `%`, `@`,
   `` ` ``.
2. **Leading indicator.** The scalar's first codepoint is `-`
   or `?` (would otherwise be parsed as a YAML 1.2 sequence /
   mapping indicator).
3. **YAML 1.2 scalar-token collision.** The unquoted form would
   resolve to a non-string YAML token. Specifically, the scalar
   - matches the *number* regex
     `^-?(0|[1-9][0-9]*)(\.[0-9]+)?([eE][+-]?[0-9]+)?$`, or
   - matches a *boolean* token under case-insensitive equality
     with exactly one of `true`, `false`, `yes`, `no`, `on`,
     `off`, or
   - matches a *null* token under case-sensitive equality with
     exactly one of `null` or `~`.
4. **Empty string.** The scalar is `""` (length zero).
5. **Non-ASCII codepoint.** The scalar contains any codepoint
   strictly greater than U+007F. Such codepoints are emitted
   inside the double quotes as `\uXXXX` escapes.

All five tests are evaluated against the *raw* scalar string before any
quoting or escaping. Inside double quotes the only escapes emitted are
`\\`, `\"`, `\n`, `\t`, `\r`, and `\uXXXX` (YAML 1.2 § 5.7); no other
escape forms are produced. If none of the five conditions holds, the
scalar is emitted unquoted.

*Cross-check, not authority.* The rule above is defined locally and **is**
the contract. It is intended to match the yarn-berry SYML writer at
[4.14.1](https://github.com/yarnpkg/berry/blob/%40yarnpkg/cli/4.14.1/packages/yarnpkg-core/sources/syml.ts)
for the working fixture set; the citation is evidence of agreement, not
authority. If a future yarn-berry patch diverges, the yarn-berry adapter
absorbs the divergence (see [§1.9](#19-acceptance-gate)); the rule above
does not move.

**Bare-emit exceptions (yarn writes these unquoted).** A small, closed set
of scalars that condition 1 (special char) or condition 3 (boolean token)
*would* quote are emitted **bare** to match yarn's own writer byte-for-byte.
The adapter applies the quoting predicate uniformly and then strips the
quotes off exactly these scalars in a post-pass:

- **`conditions`** (entry-level, 2-space indent) — a scalar platform gate
  (`os=darwin & cpu=arm64`, `(os=darwin | os=linux)`) that condition 1
  would quote for its spaces / `&` / `|`. Emitted bare.
- **`dependenciesMeta` / `peerDependenciesMeta` booleans** (the value-level
  keys `optional`, `built`, `unplugged` at 6-space indent — nesting level 3,
  reached only inside these two meta blocks) — `true` / `false` that
  condition 3 would quote as the boolean token. Emitted **bare**, exactly
  as yarn writes them. This is load-bearing, not merely cosmetic: `built:
  "false"` is a non-empty (truthy) string, so yarn's `if (meta.built)` would
  read it as `true` and may run a postinstall the lock meant to suppress;
  the bare `built: false` is the only correct emit. The unquote is scoped to
  the 6-space indent + that three-key set + a literal `true`/`false` value,
  so a genuine string field whose value happens to be `"true"`/`"false"`
  (e.g. a `bin` target at 4-space indent) is never touched.

### 1.6 Indent, line endings, trailing newline

- **Indent.** Two spaces per level (a yarn-berry invariant).
- **Line endings.** Default `lf`. `crlf` is opt-in via the `lineEnding`
  option for environments that round-trip through CRLF tooling. A CRLF
  input vs LF output is a *style* difference, not a roundtrip failure —
  the graph is unaffected.
- **Trailing newline.** Single `\n` (or `\r\n` under `crlf`). Yarn's own
  emitter always trails a newline; we match.

### 1.7 `__metadata.cacheKey` threading

Re-emitted from `options.cacheKey` if supplied; otherwise omitted. The
graph carries no `cacheKey` storage — it is a **write-time ambient
input**, not a graph-level fact (consistent with treating `cacheKey` as
lockfile-level rather than per-tarball). Roundtrip via
`parse → stringify → parse` preserves `cacheKey`-derived `checksum`
values bit-for-bit because checksums round-trip as opaque strings; the
cacheKey itself does not need to round-trip unless the caller explicitly
threads it.

### 1.8 Non-goals (explicit)

- **Byte-lossless roundtrip.** Comments, original whitespace, the
  caller's quoting style, and original key ordering MAY diverge from the
  input bytes.
- **CST-grade fidelity.** No source-map, no comment retention, no
  preservation of multi-spec entry-key concrete shape beyond what the
  graph remembers. If the input merged `foo@npm:^1, foo@npm:^2` into one
  entry, stringify emits it as one entry; if a future modifier splits the
  spec set, stringify emits two entries. Recovering the original concrete
  shape across modifier action is not in scope.
- **Field-level preservation of `__metadata` keys we do not model.**
  Unknown `__metadata` keys are dropped on parse with a `warning`
  diagnostic; stringify cannot resurrect them.

### 1.9 Acceptance gate

For every fixture under
`src/test/resources/fixtures/lockfiles/*/yarn-berry-v<N>.lock`:

1. `parse(stringify(parse(x))).diff(parse(x))` is structurally empty.
2. `parse(stringify(parse(x))).tarballs()` is iteration-equal to
   `parse(x).tarballs()`.
3. Output passes a SYML well-formedness check (re-parsing succeeds
   without a SYML parse error).
4. Output under `lineEnding: 'crlf'` differs from the `lf` form only in
   line endings.

New fixtures join the gate's working set automatically. Each yarn-berry
spec may attach a *Known degradations* note that scopes the gate to the
graph-level facts its parser currently extracts.

## §2 Patch slot & TarballKey sentinel

### 2.1 The `+patch=` slot

`TarballKey` extends from a flat `${name}@${version}` to a *suffixed*
string with ordered disambiguator slots. The unsuffixed form remains
canonical for the common case; slots appear only when an adapter supplies
a slot value. The full grammar is in [§4.3](#43-tarballkey). The one slot
defined today is `patch`:

- For **file-backed** patches, the `+patch=` value is the **sha512 of the
  patch source bytes**, lower-case hex, no prefix. The adapter computes it
  itself — it does **not** copy a value the PM happened to record.
- For yarn **built-in** patches (`~builtin<compat/…>`, which have no file
  on disk and ship different bytes under the same locator across yarn
  major versions), the canonical source is `<yarn-major>:<locator>` (e.g.
  `4:~builtin<compat/typescript>`); the fingerprint is the sha512 of those
  UTF-8 bytes, lower-case hex. Two recipes, one shape.

Both recipes are 128 lower-case hex characters; never empty, never
carrying `+` or whitespace, never the `unresolved-` prefix (reserved for
the sentinel — see [§2.2](#22-the-unresolved-sha256-sentinel)).

### 2.2 The `unresolved-<sha256>` sentinel

When the recipe input is **not present at parse time** (a CI artefact
stripped of `.yarn/patches/`, a slim source tarball, a hand-edited entry),
the adapter emits a `warning` diagnostic and assigns the sentinel
fingerprint `unresolved-<sha256-of-input>` — the literal prefix
`unresolved-` followed by exactly 64 lower-case hex characters (sha256, to
distinguish it visually from the canonical sha512 form). The sentinel
input is **per-format and adapter-defined**, in deliberate contrast to the
canonical recipe (which is one shape across PMs):

| Format | Sentinel input (UTF-8 bytes hashed with sha256) |
| --- | --- |
| yarn-berry (file / `~builtin<…>`) | the locator string verbatim, e.g. `patch:lodash@npm%3A4.17.21#./patch.diff::version=4.17.21&hash=…` |
| pnpm | `<name>@<version>:<patched_dependencies-key>` where `<patched_dependencies-key>` is the literal map key as written in `pnpm-lock.yaml` (e.g. `lodash@^4.0.0`, `lodash`) |

The sentinel keeps `parse → stringify` round-trips intact and forces
*input* divergence — two unreachable patches on the same `name@version`
with different sentinel inputs do not collide. Because the sentinel input
is PM-native, the sentinel namespace is **explicitly degraded**: two
adapters reading the same artefact with different input encodings produce
different sentinels, and that is intentional — **the sentinel does NOT
participate in cross-PM identity**.

The `unresolved-` prefix is reserved spec-wide and is the spec-defined
predicate (`patch.startsWith('unresolved-')`) on which mutator-class
operations refuse (see [mutator-coherence in §4.5](#45-mutator-coherence)).
Read-only flows (parse, stringify, diff) are unaffected, and
`removeTarball` against a sentinel key is permitted; only modification is
barred.

### 2.3 Canonical-vs-PM-native attribution principle

> **For any cross-PM type, the canonical form is owned by this layer.
> PM-native primitives are *attribution* — recorded for forensics, never
> load-bearing for identity, dedup, comparison, or storage keys.**

Applied to patches: different PMs hash different things and call all of it
`sha512` — yarn-berry's `hash=` is the sha512 of the *patched archive
bytes*; pnpm's `patched_dependencies` records a file path with no digest
at all. Adopting either verbatim breaks cross-PM dedup (the same
`lodash@4.17.21` + same `.patch` file would land under two different
`+patch=` fingerprints after a yarn → pnpm migration). One canonical
recipe owned by this layer is the **necessary** condition for two adapters
to produce the same key for the same artefact — the load-bearing property
for `audit-fix`-class mutations across mixed projects. The PM-native
`hash=` / `patched_dependencies` value is recorded as **attribution
only** (diagnostics, forensic round-trip, per-format passthrough); it is
never consulted for identity, equality, or dedup, and it never enters the
core Graph types. Whether a given primitive survives round-trip depends on
what the format adapter chooses to passthrough, not on the cross-PM model.

(Sufficiency in the cross-PM case additionally requires byte normalisation
across format boundaries — CRLF, trailing newlines, archive packaging —
which is a separate, layer-owned concern and not part of the slot
recipe.)

## §3 Integrity model

Graph-canonical integrity is a structured, loss-free carrier — an ordered
**hash multiset** with a per-hash **origin tag** — **not** a single
sha512. Integrity is `TarballPayload` data; it is **identity-neutral** —
NOT part of NodeId or TarballKey.

```ts
type HashOrigin = 'sri' | 'berry-zip' | 'url-fragment' | 'registry' | 'recomputed'
interface Hash      { algorithm: string; digest: string /* canonical lowercase hex */; origin: HashOrigin }
interface Integrity { hashes: Hash[] }   // verbatim multiset, source order
// (multi-hash "set-ness" is captured by multiset cardinality — two origin:'sri'
//  hashes re-emit as a space-joined SRI — so no per-hash `sri-multi` tag is needed.)
```

### 3.1 Preserve every algorithm verbatim

On parse, every algorithm present on disk is preserved verbatim — `sha1`,
`sha256`, `sha384`, `sha512`, **every** member of a space-joined
multi-hash SRI, and the yarn-classic `resolved#<40hex>` sha1 fragment.
Nothing is dropped; `sha1-` is accepted. (A space-joined `sha1-… sha512-…`
SRI is preserved in full — including its recoverable sha512 — rather than
collapsed to one member.)

### 3.2 Origin tags — the load-bearing addition

`origin` distinguishes a **tarball** sha512 (`origin ∈ {sri, registry,
recomputed}` — re-encodable into an SRI) from a **berry-zip** sha512
(`origin = 'berry-zip'` — re-encodable only within the yarn family). This
makes the derive-vs-fetch boundary *representable* instead of silently
mis-converted (the boundary itself is stated in
[§3.3](#33-the-berry-zip--tarball-sri-boundary)). The five origins:

| Origin | Meaning |
| --- | --- |
| `sri` | digest parsed from a Subresource-Integrity field (npm / pnpm / bun / yarn-classic) |
| `berry-zip` | sha512 over Yarn Berry's post-processed **zip-cache** representation (the `checksum:` field) — NOT a tarball digest |
| `url-fragment` | sha1 from a yarn-classic `resolved#<sha1>` URL fragment (origin reserved; in current phasing this sha1 is carried on the resolution sidecar rather than folded into the multiset) |
| `registry` | digest taken from registry metadata (e.g. `dist.integrity`) |
| `recomputed` | digest recomputed from fetched tarball bytes |

### 3.3 The berry-zip ≠ tarball-SRI boundary

Yarn Berry's `checksum` is **not** the tarball's sha512; it is a digest of
yarn's post-processed zip-cache. Verified on `lodash@4.17.21`: the pnpm
SRI decodes to hex `bf690311…`, the berry checksum is `d8cbea07…` —
**different digests of different byte streams.** They are therefore **not
interchangeable**:

- A **tarball-origin** sha512 MUST NOT be written into a berry `checksum`.
- A **berry-zip** sha512 MUST NOT be written into an SRI field.

Within the yarn family, a berry checksum *does* round-trip (berry↔berry is
a pure hex re-encode of the same zip-cache digest — raw hex pre-v8,
`<cacheKey>/<hex>` v8+; the cacheKey prefix is sidecar attribution).

### 3.4 Omit, never fabricate

Emit is **origin-aware**. SRI formats (npm / pnpm / yarn-classic / bun)
emit all members of the multiset. Yarn-berry `checksum` emits **only** a
`berry-zip` sha512. When the target needs an origin/algorithm that is not
present in the carrier, the adapter emits the **best available** digest
and a soft `RECIPE_INTEGRITY_INCOMPLETE` diagnostic — **never a digest the
target PM would reject.**

The canonical case: converting npm / pnpm → yarn-berry, no berry-zip
digest is derivable offline (yarn hashes the zip-cache, not the tarball),
so the `checksum:` line is **omitted entirely** — a valid yarn lock, which
yarn re-computes on install — accompanied by `RECIPE_INTEGRITY_INCOMPLETE`.
"Best available" for a berry checksum slot from a tarball-only source is
*nothing*; a registry/tarball sha512 is never substituted. This is the
omit-not-fabricate posture: a missing checksum that yarn self-heals beats
a wrong one it rejects.

**Comparison digest.** When a tarball-origin sha512 is present it is the
canonical compare digest; otherwise the strongest shared algorithm.
Identity is unaffected (integrity is not in the NodeId / TarballKey), so a
cross-family comparator MUST compare the multiset and respect `origin`
rather than doing presence-gated string-equality over a single digest.

## §4 Reserved vocabulary

The minimum model vocabulary a format spec relies on. These definitions
are scoped to what the format specs reference; the full graph / API model
is broader.

### 4.1 NodeId

A graph node is a **package instance**, not a package version: two
instances of the same `name@version` are distinct nodes when their
peer-dependency context differs.

```
NodeId      := <name>@<version>(<peerContext>)?
peerContext := (peerId)+                          # one or more, concatenated
peerId      := <name>@<version>(<peerContext>)?   # recursive
```

(Borrowed verbatim from pnpm's package-id grammar, lockfileVersion 6+.)

- **No peers** → just `name@version` (e.g. `lodash@4.17.21`).
- **One or more peers** → a parenthesised list, **sorted alphabetically by
  peer name** (e.g. `react-dom@18.0.0(react@18.0.0)`).
- **Sub-peers nest** to the depth they are resolved at (e.g.
  `apollo-client@3.7.0(graphql@16.6.0)(react@18.0.0(react-dom@18.0.0))`).
- **Scoped names** keep their `@`; the version is what follows the *last*
  `@` at the current depth.
- **Unfulfilled peers** are **omitted** from the parens. The fact that a
  declared peer was not satisfied is recorded as a diagnostic, not encoded
  into the NodeId.
- **Peer cycles** are broken by replacing the cycling sub-id with `*`
  (e.g. `a@1.0.0(b@1.0.0(a@*))`); this applies to peerContext recursion
  only.

The serialization is **deterministic**: alphabetical sort of peer-name
groups, canonical lower-case, exactly one `(...)` block per peerId — no
whitespace, no empty parens. `id` is a pure function of `(name, version,
peerContext)`.

Formats that flatten peer-virtualization (`npm-*`, `yarn-classic`) cannot
encode distinct NodeIds that share `name@version`; emitting to them
collapses instances (pick one, drop the rest, diagnostic per dropped
instance — see each format's `## Degradation rules`). Yarn-berry's own
`virtual:<hash>#…` form is the opaque alternative this readable grammar
replaces.

### 4.2 peerContext

The structured peer list embedded in a [NodeId](#41-nodeid). It and the
graph's `peer` edges are **two views of the same fact** and stay in sync:
for every peer
in `peerContext` there is a `peer` edge from this node to that peer's
NodeId, and vice versa. `peerDependenciesMeta.optional` is an **attribute**
on a `peer` edge (`{ optional: true }`), not a distinct edge kind; its
value originates in the *parent* package's
`package.json#peerDependenciesMeta`.

### 4.3 TarballKey

A deterministic string keying cross-format artefact metadata
(`TarballPayload`) on the graph. It is the NodeId stripped of peer-context,
plus ordered disambiguator slots.

```
TarballKey      := <name>@<version> ( '+' <slot-name> '=' <slot-value> )*

slot-name       := 'patch'                 # the only slot today; future slots are ADR-gated
slot-value      := canonical-token | sentinel-token
canonical-token := per-slot fixed shape; for `+patch=`, sha512 of the
                   canonical recipe input bytes — 128 lower-case hex
                   chars; never empty, never carrying `+` or whitespace,
                   never the `unresolved-` prefix
sentinel-token  := literal `unresolved-` followed by exactly 64 lower-case
                   hex chars (sha256 hex)
```

Peer-virtualised siblings that share `name@version` and any slot inputs
share **one** entry (five virt-copies of `react@18.0.0` reference the same
`TarballPayload`). The `+patch=` recipes and the sentinel inputs are
specified in [§2](#2-patch-slot--tarballkey-sentinel).

`toTarballKey(inputs)` is the **only** route to a `TarballKey` value: it
throws on a malformed slot value (`+` or whitespace, empty string, or a
value matching neither token form), drops `undefined` slots, sorts
remaining slots alphabetically by slot-name, and concatenates. Equal
inputs always yield byte-equal keys (deterministic, idempotent); callers
do not construct key strings by concatenation. `Node.patch` is the
per-node carrier for the `+patch=` slot value; it describes the underlying
tarball, not the per-instance peer context.

### 4.4 Graph

The L2 model object: package instances (nodes) connected by directed,
kind-labelled edges, plus a tarball surface
`Map<TarballKey, TarballPayload>`. Edge kinds:

| Kind | Direction | Notes |
|------|-----------|-------|
| `dep` | parent → child | regular `dependencies` |
| `dev` | parent → child | only present on workspace nodes |
| `optional` | parent → child | resolution may legitimately fail |
| `peer` | parent → expected-child | resolved against ancestor scope |

**Workspaces** are nodes with a known layout position
(`Node.workspacePath !== undefined`, where the empty string is the root
workspace) and are roots of the non-workspace subgraph. Cross-workspace
`dep` / `dev` / `optional` / `peer` edges between workspace nodes are
permitted and do not strip the target of its workspace nature; only a
*published* (registry / tarball / git) source depending on a workspace is
rejected at seal — see published
[ADR-0017](../decisions/0017-graph-seal-workspace-edges.md).

A `Lockfile` is the public alias for a sealed `Graph`. **Iteration order**
is canonical and content-derived: collections iterate in lexicographic
order of `(name, version, peerContext, kind)`. Format emitters apply their
own write-order on top — the graph's iteration is the deterministic
*input*, not the final emit order.

### 4.5 mutator-coherence

The rule that a node's `TarballKey` and its `TarballPayload` stay
consistent across mutation. `replaceNode(id, newNode)` whose `name@version`
or any slot input (`patch`) differs from the replaced node describes a
*different tarball*; the caller must pair it with the matching
`setTarball` / `removeTarball`. Stale tarball entries are **not**
auto-pruned — modifiers own lifecycle. `replacePeerContext` requires no
tarball mutation (same `name@version`, same slot inputs, same key, same
payload).

**Sentinel entries refuse mutation.** A node whose `patch` matches the
sentinel-token form (`patch.startsWith('unresolved-')`) is **read-only**
at the mutator layer: `replaceNode` of the patched node, `setTarball`
against the sentinel key, and any `audit-fix`-class operation against it
throw an `IRREDUCIBLE_LOSS` error — the canonical recipe never ran, so the
graph cannot tell whether two sentinel-keyed siblings reference the same
artefact and a mutation could silently fork them. `removeTarball` against a
sentinel key is permitted (deletion cannot fork siblings). Read-only flows
(parse, stringify, diff) are unaffected.

### 4.6 Diagnostic and the `RECIPE_*` code family

A graph and a stringify pass carry diagnostics. Wire shape:

```ts
type Diagnostic = {
  code:     string                       // discriminant, e.g. 'RECIPE_INTEGRITY_INCOMPLETE'
  subject?: string                       // NodeId or workspace path
  severity: 'info' | 'warning' | 'error'
  message:  string
}
```

Sealing rejects `error`-severity diagnostics; modifiers that cannot
satisfy a constraint emit `warning`-severity rather than blocking. The
`code` set is the abstract contract; the binding chooses the host type
(TypeScript raises a single `LockfileError` discriminated by `code` —
e.g. `INVALID_INPUT`, `IRREDUCIBLE_LOSS`, `CAPABILITY_LACK`,
`INVARIANT_VIOLATION`).

The **`RECIPE_*` family** is the set of soft diagnostics a format adapter
emits when a fact held in the graph cannot be faithfully represented in
the target format and a degradation rule applies (rather than a hard
throw). The format specs reference these members:

| Code | Severity | `subject` | When | Posture |
| --- | --- | --- | --- | --- |
| `RECIPE_INTEGRITY_INCOMPLETE` | warning | the affected node | the target integrity slot needs an origin/algorithm the carrier lacks (notably: → yarn-berry `checksum` from a tarball-only source) | omit the field, never fabricate ([§3.4](#34-omit-never-fabricate)) |
| `RECIPE_PEER_META_INCOMPLETE` | warning | the consumer node | an optional-peer flag was requested via an external lookup that no source could answer | omit the marker, never guess |

The omit-not-fabricate posture is shared across the family; members differ
in *when* they fire. `RECIPE_INTEGRITY_INCOMPLETE` fires whenever a held
integrity fact cannot be represented; `RECIPE_PEER_META_INCOMPLETE` fires
only when an external rung was explicitly requested and failed (in the
default offline mode the graph is the sole authority and the pass is
silent). Individual format specs define any further `RECIPE_*` members
they emit.

## §5 Descriptor→node resolution (yarn-family parse)

> **Scope.** §5 is the **yarn-family parse** contract — it governs how a
> consumer's dependency *descriptor* is matched to a lockfile *entry* (node)
> during `parse(input, options?)` for **yarn-classic** and the **yarn-berry**
> family (v3–v10). npm / pnpm / bun do **not** use this ladder: their lock
> bodies pre-resolve every edge to a concrete entry id, so there is no
> descriptor→entry matching step to perform at parse.

### 5.1 Why a ladder is needed

The yarn lock records **ranges**, not resolved versions: a consumer's
`dependencies` block holds the verbatim *descriptor* it declared
(`csstype: "npm:^3.1.3"` in berry; `csstype "^3.1.3"` in classic), and the
**entry key** is the join surface — yarn keys each entry by the descriptor(s)
that resolve to it (`"csstype@npm:^3.1.3":` / `"csstype@^3.1.3":`). Parse
binds an edge by looking the descriptor up in an index of entry keys.

A manifest **override** (npm `overrides`, yarn `resolutions`, pnpm
`pnpm.overrides` — the canonical PM-neutral form is specified in published
[ADR-0025](../decisions/0025-manifest-overrides.md)) breaks that exact-match
join. When a dependency is forced, yarn **rewrites the entry key to the
pinned descriptor** and drops the consumer's own range from the key. For a
backstage-style `"csstype@npm:^3.1.3": "3.0.9"` resolution the entry becomes:

```
"csstype@npm:3.0.9":          # key = the PIN, not the consumer's ^3.1.3
  version: 3.0.9
  resolution: "csstype@npm:3.0.9"
```

A consumer still declaring `csstype: "npm:^3.1.3"` now finds **no** entry
keyed `csstype@npm:^3.1.3`. Two further facts make a naive fix unsafe:

- **The pin may be NON-satisfying.** `3.0.9` does **not** satisfy `^3.1.3`,
  so pure semver cannot recover the link — the override map is *mandatory*
  for this class.
- **The pin may target a non-version.** A `resolutions` value can be a
  `patch:` / `portal:` / git descriptor, not a registry version, so the
  resolver must re-feed the pin's target through the *whole* ladder rather
  than assume a semver point.

yarn writes **no lock-borne resolutions** — the override declaration lives in
`package.json` only. So the override map exists at parse time **iff the caller
passed `manifests`** (the public `parse(format, input, { manifests })`
F6-captures them per [ADR-0025](../decisions/0025-manifest-overrides.md) into
the canonical override form *before* the adapter parse, and threads the
constraints into the edge resolver).

### 5.2 The resolution ladder (normative)

Each consumer descriptor is resolved by the rungs below **in order**. Rung 0
is the steady-state path; Rungs 1–3 run **only** when Rung 0 misses, so the
common case stays a single O(1) lookup. The first rung that yields a node
binds the edge; the dropped-edge set this ladder produces is a **subset** of
the exact-match-only result (it only ever *shrinks*).

| Rung | Rule |
| --- | --- |
| **0 — exact** | `index.get("<name>@<descriptor>")` — exact entry-key string match. Unchanged, first, O(1). |
| **1 — structural fallbacks** (berry only) | A `patch:` descriptor binds to its patch node after the entry-key's `::version=…&hash=…[&locator=…]` param block is stripped; a `link:` / `portal:` descriptor binds via the `::locator=<consumer>` qualifier reconstructed from the consumer's own resolution. More-specific than the rungs below, so it runs first. |
| **2 — override map** | If an override constraint matches `(depName, declaredRange, consumerPath)`, resolve its `to` target by feeding `to` **back through Rungs 0+1** (`to:"3.0.9"` → the exact `<name>@npm:3.0.9` node; `to:"patch:…"` → the patch node). This handles the NON-satisfying pin. The override is **authoritative** — a human declaration beats inference — so it precedes semver. |
| **3 — source-gated semver** | For an `npm:`/bare descriptor **only**, among the entries that share `depName` and whose **source is a registry tarball**, pick the **max-satisfying** locked version. A single eligible registry sibling binds directly (the unique pin yarn chose, even if its version does not satisfy the bare range). A final tie at the maximum → **diagnose and drop**, never guess. |
| **4 — drop** | No rung matched → the edge is dropped with the adapter's `*_UNRESOLVED_DEP` / `*_MISSING_ENTRY` diagnostic. Unchanged. |

### 5.3 The source-awareness invariant

Rung 3 is **source-gated**: an `npm:`/bare descriptor may bind **only** a
node whose canonical resolution is a registry **tarball**
([§4.4](#44-graph) source taxonomy; the four-case canonical resolution is
specified in published
[ADR-0014](../decisions/0014-canonical-recipe-input-normalisation.md) §4.F3
— `tarball` / `git` / `directory` / `unknown`). A git-fork, directory-link,
`unknown`-source, or resolution-less node is **invisible** to a registry
range: a fork pinned at a satisfying version must never silently satisfy an
`npm:` descriptor, because the bytes are not the registry artefact the range
asked for. An `unknown`-source node is likewise ineligible — the resolver
cannot *prove* it is a registry tarball. This gate is independent of node
**identity**: it holds regardless of whether the source class is also folded
into the NodeId, so the ladder stays correct on its own.

### 5.4 Diagnostics

Both yarn adapters emit, version-prefixed (mirroring the
`<prefix>_PEER_AMBIGUOUS` family):

| Code | Severity | `subject` | When | Posture |
| --- | --- | --- | --- | --- |
| `<prefix>_AMBIGUOUS_RESOLUTION` | warning | the consumer node | Rung 3 found ≥2 candidates tied at the maximum satisfying version | drop the edge, never guess (mirrors `<prefix>_PEER_AMBIGUOUS`) |
| `<prefix>_RESOLUTION_PIN_UNRESOLVED` | info | the consumer node | an `npm:`/bare descriptor missed Rung 0 **and** no semver candidate satisfied it, with **no** `manifests` supplied | point at the missing override source — the signature of a `resolutions` pin to a non-satisfying descriptor |

**Manifest-less limit.** Without `manifests`, only Rung 3 runs for forced
deps. It recovers the *satisfying* slice (a `^3.0.2` consumer still finds a
satisfying registry sibling). A **non-satisfying** pin cannot be recovered —
it either stays dropped (no satisfying sibling → `*_RESOLUTION_PIN_UNRESOLVED`
fires) **or**, when a different satisfying sibling exists, binds to *that*
sibling rather than the pin (Rung 3's best inference; no diagnostic, since the
edge resolved). Supplying `manifests` is the only way to honour a
non-satisfying pin faithfully. This is the documented degradation, not a bug:
the lock alone does not carry the pin.
